REACTIVE TODOS

Another iteration of the ubiquitous todo application—hopefully, this time, it can be called a finite instance.

This application is the result of a process of exploration. I searched for synergy while implementing a web application using the Quarkus framework for the server-side and the HTMX library for the client-side.

The reactive nature of Quarkus led me into the world of Mutiny, where I encountered a plethora of tasks and changes that reactive programming implies, such as unis, multies, blocking, and non-blocking operations. Honestly, I realized that I prefer imperative programming more.

Server-Sent Events (SSE) are used to push back synchronous events from the server.

Being a victim of enterprise Java, I followed the Ports & Adapters architecture, incorporating both clean and non-clean code. The project is sharded into modules, making it easy to split it into microservices or simply implement a new version and migrate away.

Finally, the light of Java 21 shines with sealed types and records, along with pattern matching—a game-changer that has altered the way I approach typical tasks. With Rust-like wrappers such as Result and Option, they have changed the way I handle error handling and optionality.

Well, I hope you gain some knowledge and inspiration here. It's been a pleasure to create this. Enjoy!
